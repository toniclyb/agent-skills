# Project Governance

## Purpose

Keep projects legible, auditable, and easy to evolve. Use this skill before creating files, moving scripts, adding docs, changing project rules, or reviewing a repo that feels disorganized.

## Core Rules

1. Separate permanent rules from task state.
2. Keep generated/runtime files out of source control.
3. Put every script in an owned location with a clear lifecycle.
4. Maintain a small project index or equivalent navigation map.
5. Require facts, tests, and review gates before code changes.
6. Prefer small, reversible changes over broad cleanup.

## Workflow

### 1. Inspect Before Writing

Run a shallow structure scan before adding or moving files:

```bash
pwd
git status --short 2>/dev/null || true
find . -maxdepth 3 -type f | sed 's#^\./##' | sort | sed -n '1,220p'
```

If available, run the bundled hygiene checker:

```bash
python ~/.codex/skills/project-governance/scripts/audit_project_hygiene.py .
```

### 2. Classify Content

Use this routing by default:

| Content | Default location |
| --- | --- |
| Agent rules and hard boundaries | `AGENTS.md` |
| Documentation map | `docs/DOCS_INDEX.md` |
| Current architecture facts | `docs/architecture/current.md` |
| Development workflow and gates | `docs/governance/` |
| Task plans | `docs/plans/` |
| Audits, incidents, evidence | `docs/audit/` |
| Research and experiments | `research/` |
| Runtime state and logs | `data/`, `logs/` |
| Reusable scripts | `scripts/` |
| Temporary scripts | `scripts/tmp/` or delete before handoff |
| Tests | `tests/` |

For detailed standards, read `references/project-standards.md`.

### 3. Keep AGENTS.md Small

`AGENTS.md` should contain stable rules only:

- project goal
- hard architecture boundaries
- required reading order
- development gates
- document routing rules
- review/audit requirements

Do not put task plans, dated audit reports, current parameter tables, large directory trees, incident narratives, or command dumps in `AGENTS.md`.

### 4. Manage Scripts Deliberately

Before adding a script, decide:

- Is it reusable? Put it in `scripts/` with a descriptive name.
- Is it one-off? Put it in `scripts/tmp/` or delete it before handoff.
- Is it deterministic governance logic? Prefer a script over prose.
- Does it touch production state? Require explicit docs, tests, and a dry-run path.

### 5. Update Indexes and Git

If a repo has an index generator, run it before and after meaningful changes. If no local git exists, initialize one only when appropriate for the user/project and ensure `.gitignore` excludes secrets, runtime data, caches, logs, dependencies, generated reports, and temporary files.

### 6. Handoff

End with evidence:

```markdown
## Handoff
- Changed files:
- Behavior changed:
- Validation:
- Not run:
- Remaining risks:
```

## Project Hygiene Checklist

- `AGENTS.md` is concise and contains only stable rules.
- `docs/DOCS_INDEX.md` exists for projects with more than a few docs.
- Task-state docs live under `docs/audit/` or `docs/plans/`.
- Runtime files are ignored by git.
- Temporary scripts are absent or clearly isolated.
- Generated files are marked and regenerated by scripts.
- Tests exist for behavior-changing governance automation.
- `git status --short` is explainable at handoff.
